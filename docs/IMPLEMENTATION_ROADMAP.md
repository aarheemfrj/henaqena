# خريطة تنفيذ هنا قنا

**الهدف:** إطلاق تطبيق iOS وAndroid ومنصة ويب/إدارة حقيقية تخدم مدينة قنا، وتعمل من مصدر بيانات واحد: PostgreSQL وAPI مشتركة.

**قاعدة التنفيذ:** كل مرحلة لا تعتبر منتهية قبل اختبارها محلياً، تسجيلها في `PROJECT_ACTIVITY_LOG.md` باسم المنفذ، وعمل كوميت مستقل لها.

## الوضع الحالي

المتاح بالفعل:

- تطبيق Flutter يعمل كنسخة أولية على iOS، ويقرأ الدليل والفئات من API.
- API مبني على Express وPrisma، وقاعدة PostgreSQL منفصلة باسم `henaqena`.
- منصة Next.js في `apps/web` على المنفذ الثابت `3100`، ولوحة أولية للإدارة.
- حساب بكلمة مرور، جلسة تطبيق محفوظة محلياً، وإضافة نشاط بعد تسجيل الدخول.
- رفع صور النشاط إلى الخادم محلياً وتخزين روابطها في PostgreSQL.
- اعتماد/رفض النشاط من منصة الإدارة، ثم ظهوره في الدليل العام.

ما زال تجريبياً أو غير مكتمل: تسجيل الدخول الاجتماعي وOTP الحقيقي، الخرائط، رفع الصور إلى تخزين إنتاجي، إدارة جميع أنواع المحتوى، الإشعارات الفعلية، الإعلانات المحلية، التقييمات الكاملة، التشغيل على VPS، ومراجعة الجودة قبل المتاجر.

## المرحلة 0 — تثبيت بيئة التشغيل والإصدارات

### المطلوب

1. اعتماد هيكل الخدمات النهائي:
   - `apps/mobile`: Flutter، iOS أولاً ثم Android.
   - `apps/api`: API مشتركة على المنفذ `4000` داخلياً.
   - `apps/web`: منصة عامة وإدارة Next.js على المنفذ `3100`.
   - PostgreSQL: قاعدة مستقلة باسم `henaqena` فقط.
2. إنشاء ملفات بيئة إنتاج واضحة وغير قابلة للرفع إلى Git.
3. استبدال `prisma db push` في الإنتاج بـ migrations مرقمة وقابلة للتكرار.
4. إعداد seed اختياري خاص بقنا لا يختلط ببيانات الإنتاج.
5. توحيد أسماء المتغيرات: رابط API العام، مفتاح الإدارة، سر الجلسات، التخزين، مزود الرسائل، خرائط، إشعارات.

### معيار القبول

- جهاز جديد أو VPS يستطيع تشغيل API والمنصة بملف بيئة فقط.
- لا يوجد سر أو كلمة مرور داخل المستودع.
- لا تلامس قاعدة مال سوفت أو أي قاعدة أخرى.

## المرحلة 1 — الهوية والحساب الحقيقي

### التطبيق

1. إنهاء شاشات إنشاء الحساب وتسجيل الدخول الحالية وربط الأخطاء العربية الصحيحة.
2. صفحة الحساب: الاسم، صورة اختيارية خاصة، البريد الاختياري، المناطق الافتراضية، الاهتمامات، الخصوصية، والإشعارات.
3. إضافة تسجيل خروج وحذف الجلسة وإدارة الجلسات المنتهية.
4. وضع الزائر: تصفح فقط؛ لا يمكنه إضافة نشاط أو إعلان أو تقييم أو بلاغ قبل التسجيل.
5. حفظ اختيارات الترحيب في الحساب عند التسجيل أو بعده.

### التحقق

1. OTP عبر WhatsApp ثم SMS كبديل، والبريد كقناة اختيارية.
2. إعادة تعيين كلمة المرور عبر قناة مختارة.
3. Google على Android وSign in with Apple على iOS بعد تجهيز حسابات المطور ومفاتيحها.

### معيار القبول

- المستخدم يستطيع التسجيل، الدخول، الخروج، استرجاع الحساب، ثم متابعة مساهماته من جهاز آخر.
- كل عملية حساسة في API تعتمد على session حقيقية، لا على header قابل للتزوير.

## المرحلة 2 — دليل «مين؟» وخدمة المكان الحقيقية

### البيانات

1. استكمال نموذج مقدم الخدمة: الاسم، الوصف، الفئات، العنوان، المنطقة، إحداثيات الخريطة، الهاتف/WhatsApp، مواعيد العمل، محلي/أونلاين، نوع الرقم، الصور (1–10)، وحالة التوثيق.
2. إضافة خدمة أو أكثر داخل مقدم الخدمة: اسم الخدمة، وصفها، هل يظهر سعرها أم لا، وعروضها.
3. فهرسة البحث العربي والإنجليزي بالاسم والفئة والمنطقة والخدمة.

### التطبيق والمنصة العامة

1. بحث شامل يوجه المستخدم للقسم أو النتيجة الأنسب.
2. تصفية بالمنطقة والفئة، مع ترتيب الأقرب عند توفر الموقع وبعد الحصول على الإذن.
3. صفحة مكان كاملة: معرض صور سحب، بيانات الاتصال، خريطة، مواعيد، الخدمات، العروض قبل الأسعار، تقييمات، وإضافة تقييم أسفل التفاصيل.
4. اتصال وفتح WhatsApp والخريطة كرابط عمل حقيقي.
5. استبدال صور المعاينة بصور النشاط المخزنة فعلياً.

### معيار القبول

- يمكن إدخال نشاط من الإدارة أو المجتمع، مراجعته، نشره، العثور عليه بالبحث، ثم الاتصال به أو الوصول إليه على الخريطة من iOS وويب.

## المرحلة 3 — إدارة النشاط من المجتمع والملكية

### التطبيق

1. «أضف نشاط» للمستخدم المسجل فقط.
2. معاينة قبل الإرسال، تحقق فوري من الحقول، منع التكرار بالاسم/الهاتف/المنطقة.
3. «أملك نشاط» و«أبلغ عن نشاط» مع مسار منفصل لكل منهما.
4. «مساهماتي»: نشاطاتي، طلبات الملكية، البلاغات، حالتها، سبب الرفض، وإعادة الإرسال بعد التعديل.

### المنصة

1. طابور أنشطة معلقة مع صورها وبياناتها كاملة.
2. اعتماد، رفض مع سبب، طلب تعديل، ودمج اقتراح مكرر.
3. سجل تدقيق: من اتخذ القرار ومتى ولماذا.
4. شارة «مضاف من المجتمع» و«موثق» وفق قواعد واضحة، من دون تكرار بصري.

### معيار القبول

- كل تعديل أو قرار يمكن تتبعه، وصاحبه يتلقى النتيجة ويرى حالته داخل التطبيق.

## المرحلة 4 — «عندك؟» الإعلانات المحلية

### قواعد المنتج المعتمدة

1. إعلان مستخدم أو صاحب نشاط بعد مراجعة الإدارة.
2. اسم، فئة، وصف، سعر إجباري، وصورة حقيقية واحدة على الأقل وحتى 5 صور.
3. مدة العرض 7 أيام؛ قبل الانتهاء إشعار إعادة نشر.
4. عند رفض التجديد أو عدم الرد 3 أيام: حذف الصور والبيانات العملية من الواجهة؛ يحتفظ المستخدم بسجل نصي للحساب.
5. إعجاب وتفاعل لتعزيز الظهور، من دون رسائل خاصة.
6. تعليق عام وردود متفرعة على التعليق.

### المنصة

1. طابور مراجعة الإعلانات، أسباب الرفض، أرشفة، وإعادة نشر.
2. فلتر المنطقة والفئة والسعر والحالة.
3. أدوات مكافحة السعر الوهمي والمحتوى المكرر والصور غير الحقيقية.

### معيار القبول

- مستخدم يسجل إعلاناً كاملاً، يراجعه المدير، يظهر لمدة صحيحة، يتفاعل الناس معه، ثم ينتهي أو يعاد نشره آلياً.

## المرحلة 5 — التقييمات والمجتمع والسمعة

### التقييمات

1. تقييم خدمة المكان بنجوم منفصلة: الجودة، الالتزام، السعر؛ ثم متوسط واضح.
2. تعليق اختياري مع صورة مستخدم أو أول حرف من اسمه.
3. شكل التعليقات مثل Facebook/Instagram: تفاعل، رد، ورد متداخل، لا بطاقات ثقيلة.
4. بلاغ عن تعليق، إخفاء/حذف إداري، وسجل للبيانات المحذوفة.

### المجتمع

1. ملف المستخدم: مساهماته، تقييماته، إعلاناته، ونقاطه، مع خيار جعل صفحة الملف خاصة.
2. المستويات المعتمدة:
   - قناوي: 0–50 نقطة.
   - قناوي رايق: 50–100 نقطة.
   - قناوي أصيل: 100–500 نقطة.
3. كل تقييم صالح يمنح نقطة؛ لوحة الإدارة تحدد المكافآت والقواعد لاحقاً.

### معيار القبول

- التقييم مرتبط بحساب حقيقي، لا يمكن انتحال صاحبه، ويمكن للإدارة معالجة الإساءة بدون فقدان سجل المراجعة.

---

## المرحلة 6 — «بكام؟» و«دلوقتي» والإعلانات الرئيسية

### «بكام؟»

1. خدمات وأسعار مرجعية، مع العروض قبل الأسعار داخل صفحة المكان.
2. اقتراح سعر من المستخدمين عند وجود خدمة محددة، مع متوسط لا يظهر إلا بعينة كافية.
3. إدارة الأسعار والعروض من المنصة، وصلاحية خاصة لصاحب النشاط الاحترافي لاحقاً.

### «دلوقتي»

1. أخبار محلية، فعاليات، تنبيهات، افتتاحات، ومحتوى حسب المنطقة.
2. إشعارات حسب الكل أو مناطق المستخدم فقط، مع ملخص اختياري.

### إعلانات الرئيسية

1. صورة، عنوان/وصف، اسم المُعلن، تاريخ بداية/نهاية، مناطق مستهدفة، وجهة فتح (صفحة/اتصال/WhatsApp/خريطة/رابط خارجي).
2. ترتيب ووزن ظهور رقمي مجموعهما 100%، وفترة تبديل قابلة للتحكم من المنصة.
3. الإعلان قابل للسحب في التطبيق مع انتقال ناعم؛ لا يعرض إذا انتهت مدته.
4. البطاقة الموسمية أعلى الرئيسية: تغيرها الإدارة للعيد القومي، الأعياد، والمناسبات بدون إصدار تطبيق جديد.

### معيار القبول

- المدير يضيف محتوى أو إعلاناً من المنصة ويظهر في التطبيق في الوقت والمنطقة والترتيب الصحيحين.

---

## المرحلة 7 — الإشعارات والخرائط والبحث والموقع

1. Firebase Cloud Messaging لنظام Android وAPNs لنظام iOS عبر Firebase.
2. مركز إشعارات داخل التطبيق: قراءة، تحديد الكل، تفضيلات الكل/المنطقة/الملخص.
3. مزود خرائط: نحدد Google Maps أو بديل مناسب للتكلفة قبل الدمج النهائي.
4. طلب إذن الموقع عند الحاجة فقط، مع اختيار يدوي دائم وعبارة «تحديثات قادمة» للمناطق غير المدعومة.
5. Pagination وdebounce للبحث والنتائج لتبقى سريعة مع 500 مقدم خدمة وما بعده.

### معيار القبول

- الإشعار يصل على هاتف حقيقي ويقود إلى المحتوى الصحيح، والبحث والخريطة لا يبطئان التطبيق.

---

## المرحلة 8 — منصة الإدارة الكاملة

1. استبدال كلمة مرور الإدارة الموحدة بحسابات فريق وأدوار: مدير، مراجع بيانات، محرر محتوى، ومراقب.
2. صفحات مستقلة: لوحة مؤشرات، مقدمو خدمات، طلبات المجتمع، الإعلانات المحلية، إعلانات الرئيسية، العروض، المحتوى/دلوقتي، التقييمات والردود، البلاغات، الإشعارات، المستخدمون، والنقاط.
3. إدخال واستيراد CSV/Google Sheets لدفعات مقدمي الخدمة مع معاينة ومعالجة الأخطاء.
4. بحث وفلاتر وتصدير وسجل تدقيق في كل مركز حساس.
5. إعدادات المناطق والفئات والموسم والبطاقات الرئيسية من دون نشر كود جديد.

### معيار القبول

- يستطيع الفريق إدارة 500 مقدم خدمة ومحتوى التطبيق اليومي دون تعديل قاعدة البيانات أو الكود يدوياً.

---

## المرحلة 9 — التخزين والإنتاج والأمان

1. الانتقال من التخزين المحلي للصور إلى object storage متوافق S3 قبل زيادة المستخدمين؛ يحفظ API رابطاً عاماً أو موقّعاً فقط.
2. تصغير الصور، توليد thumbnails، حد أحجام وأنواع، وفحص محتوى أساسي.
3. نسخ احتياطي يومي PostgreSQL واختبار استعادة دوري.
4. HTTPS وNginx/Caddy أمام API والمنصة، مع المنصة على `3100` داخلياً دائماً.
5. مراقبة الأخطاء والأداء وسجلات منظمة وتنبيهات الأعطال.
6. CORS مقيّد بالدومينات الفعلية، rate limiting، تدوير المفاتيح، وقائمة أسرار إنتاج منفصلة.

### معيار القبول

- إعادة تشغيل الـ VPS لا تفقد الصور أو البيانات، ويمكن استعادة نسخة قاعدة البيانات واختبار الخدمة من خارج الشبكة.

---

## المرحلة 10 — الجودة والإطلاق

1. اختبارات API: التسجيل، الصلاحيات، الرفع، المراجعة، الإعلانات، التقييمات، انتهاء الصلاحية، والإشعارات.
2. اختبارات Flutter: المسارات الأساسية، الأخطاء، عدم الاتصال، واستعادة الجلسة.
3. اختبارات Next.js: الحماية، صلاحيات الإدارة، الإدخال، والمراجعة.
4. تجربة مغلقة في قنا: 20–50 مستخدماً و50–100 مقدم خدمة قبل الإطلاق العام.
5. صفحات قانونية عربية: الخصوصية، الاستخدام، حذف الحساب، وسياسة المحتوى والتقييم.
6. Apple Developer وGoogle Play Console: الأيقونات، لقطات الشاشة، وصف المتجر، سياسة الخصوصية، توقيع التطبيق، TestFlight وInternal Testing.
7. الإطلاق المتزامن على iOS وAndroid بعد اجتياز قائمة الإطلاق، لا وفق موعد ثابت.

## مسار التصميم المستمر

التصميم لا يوقف التنفيذ، لكنه يراجع قبل إغلاق كل شاشة:

1. الحفاظ على هوية هنا قنا: تركواز، ذهبي، Cairo/Tajawal، وواجهة هادئة لكن ليست فارغة.
2. الفئات في شريط أفقي مضغوط بدل شبكات طويلة.
3. الانتقالات: قصيرة، ناعمة، مشتركة المحور؛ لا حركة مستمرة بلا فائدة.
4. استخدام مراجع Pinterest المحفوظة فقط كإلهام أصلي عند قول المالك «موشن جرافيك تنقل الصفحات».
5. بطاقة الرئيسية والمواسم يتحكم بها المدير؛ لا نثبت شكلها في كود التطبيق.
6. كل تحسين بصري يأتي في كوميت منفصل بعد موافقة المالك على معاينة.
